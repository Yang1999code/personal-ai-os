# 经验库 Agent 维护规则

这个目录是一个本地私有 LLM Wiki（大语言模型维基）经验库。目标是把零散经验沉淀成可复用、可追溯、可持续更新的知识。

## 先读顺序

Agent 进入本目录后，先按这个顺序读取：

1. `index.md`：了解轻量总入口、目录地图和分类入口。
2. `维护/当前上下文交接.md`：了解当前重点主题、写入原则和持续追加规则。
3. `导航/README.md`：了解前台导航页怎么读、怎么写。
4. `维护/全量索引.md`：必要时查看完整 source/wiki 台账。
5. `wiki/README.md`：了解 wiki 页面的标准格式。
6. 相关导航页。
7. 相关分类索引。
8. 相关 wiki 页面。
9. 必要时再读 `sources/` 里的原始资料。

不要一上来全文扫描所有资料。先通过索引定位，再读最相关的内容。

## 目录职责

- `sources/`：原始经验、截图、脚本、README 等来源材料。尽量保留原貌。
- `wiki/`：从来源材料提炼出的结构化知识页。
- `templates/`：新增来源、wiki 页、分类索引、故障排查页时使用的格式模板。
- `导航/`：给人和 Agent 看的前台地图，只放入口、分类和简短说明，不放大段原始资料。
- `inbox/`：临时放入的新资料。整理时需要归档到 `sources/` 并更新 wiki。
- `index.md`：轻量总入口，给人和 Agent 快速判断去哪看。
- `维护/`：后台维护文件，包含完整台账、交接页、设计说明、更新记录和后续计划。

## 分类规范

当前固定分类：

- `codex`：Codex Desktop、Codex CLI、Codex 连接服务、浏览器读取等经验。
- `claude-code`：Claude Code、Claude Agent、Claude 相关远程操作经验。
- `daily`：Mac、终端、快捷键、文件路径等日常使用经验。
- `vm`：虚拟机、SSH、远程机器、隧道、连接工具等经验。
- `troubleshooting`：跨分类故障排查、排错清单、常见问题。

如果新经验同时属于多个分类，优先放到最稳定的基础分类。

## 新增资料流程

1. 如果用户给的是临时资料，先放入 `inbox/`。
2. 判断分类和主题名。
3. 在 `sources/<category>/<topic>/` 下创建主题目录。
4. 原始 Markdown 使用 `source.md`，图片和附件尽量放在同一主题目录或子目录。
5. 根据 `templates/source-template.md` 补充来源页元数据。
6. 根据 `templates/wiki-template.md` 新建或更新对应 wiki 页。
7. 更新根目录 `index.md`、必要时更新 `维护/全量索引.md` 和分类 `index.md`。
8. 在 `维护/log.md` 记录本次维护。

如果新增资料会影响人的查找路径，还要同步检查对应 `导航/*.md` 是否需要补入口。导航页只做索引，不要把原始资料正文搬进去。

## 更新已有资料流程

- 原始事实、截图、脚本、命令输出：放入 `sources/`。
- 提炼后的结论、流程、排查清单：更新 `wiki/`。
- 发现旧 wiki 和 source 冲突：以 source 为事实依据，更新 wiki，并在 `维护/log.md` 记录。
- 不确定的新规则：先写到 `维护/roadmap.md`，不要直接大改 `AGENTS.md`。

## 敏感信息规则

这个经验库是本地私有知识库，但仍然要控制敏感信息扩散。

处理原则：

- 不在 wiki 中复述完整 token（令牌）、key（密钥）、密码、私钥。
- 内网 URL、IP、Host、账号等必要时可以写“存在这类配置”，具体值优先留在 source。
- 如果必须保留具体值，页面 `sensitivity` 标为 `sensitive`。
- 对外分享前必须先做脱敏副本。

## 回答用户时的规则

- 尽量用中文。
- 英文专有名词后加中文解释，例如 LLM Wiki（大语言模型维基）。
- 每次回答后加白话总结。
- 复杂概念可以用 Mermaid（流程图）辅助。
- 涉及本机可验证的问题，优先检查本地状态，再回答。

## 白话总结

人找资料先看 `index.md` 和 `导航/`；Agent 写资料先看 `AGENTS.md`。`sources/` 放证据，`wiki/` 放答案，`维护/` 放日志、计划和完整台账。
