# AI Loop Mini 3 架构说明

## 一句话

AI Loop Mini 3 是一个 while-driven deep loop（while 脚本驱动的深度循环）。

它不是全自动机器人，而是在“一个会用 AI 的人”的操作流程外面，加一个很薄的 while 脚本执行层：

```text
人给目标
脚本生成 request
AI 吃完整上下文
AI 深度探索一轮
AI 自检和评价
AI 留下记忆
复杂判断返回人
最多三轮收敛
```

## 和普通 AI 对话的区别

普通对话容易变成：

```text
人问一句
AI 答一句
人忘了上轮上下文
AI 也不知道之前怎么判断
```

Mini 3 想解决的是：

- 防止人忘记步骤
- 防止上下文丢失
- 防止 AI 只做浅层回答
- 防止没有测试和评价
- 防止复杂问题硬自动化
- 防止多终端交互时读到半截文件

## 核心流程

```text
1. 人填写目标和上下文包
2. 终端 A 运行 while 脚本
3. 脚本写入 round request
4. 终端 B/C 的 Claude 或 Codex 读取 request 并写 outbox
5. 脚本读取完整输出文件
6. 判断：完成、下一轮、搜索、返回人
```

## 三轮策略

- 第 1 轮：建模和深度探索
- 第 2 轮：深入解决或验证
- 第 3 轮：收敛和交付

如果一轮已经完成，可以提前停。

如果一轮没有质的提升，先反思，再决定是补上下文、搜索、测试、返回人，还是停止。

## 关键文件夹

### `02-runtime/`

执行层。这里放 `run-loop.sh`、运行配置和锁协议。

脚本只负责轮次、文件读写和停止判断，不替 AI 思考。

### `03-exchange/`

交互层。多终端、多 Agent（智能体）通过这里交换文件。

`inbox/` 放脚本生成的 request，`outbox/` 放 AI 写回的结果，`tmp/` 和 `locks/` 用来避免半写半读。

### `04-context-pack/`

上下文包。这里放背景、已知材料、现状、约束和本轮关注点。

它的作用是减少碎片化追问，让 AI 每一轮都有足够信息。

### `06-reflection/`

反思区。当一轮没有明显推进时，AI 要在这里分析原因：

- 是目标不清？
- 是上下文不够？
- 是需要上网搜索？
- 是需要测试？
- 是需要人做判断？
- 是只是重复上一轮，没有新证据？

### `07-evaluation/`

评价区。每轮都要有测试或评价。

评价可以是硬测试，也可以是软评价。

### `08-git-backups/`

本地 git（版本控制）记录区。它记录这个 Loop 模板本身的版本变化，方便回退和对比，不代表自动推送或自动部署。

### `09-boundaries/`

返回人工规则。复杂判断返回给人，不是失败，而是正常设计。

### `12-target/`

目标区。这里写要探索或解决的问题、目标地址、Done（完成）判断和复杂度。

## 白话理解

Mini 1 更像一个“AI 自主工作台”。

Mini 3 更像一个“高手使用 AI 的操作流程 + 很薄的 while 执行壳”：

```text
准备材料
脚本发 request
AI 深挖并写回 outbox
脚本看 status
复杂处问人
三轮内收敛
```
