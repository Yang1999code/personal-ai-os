# 主 Loop

你是这个文件夹的 Loop（循环）执行者。核心原则是：最大程度相信 AI 的判断，让 AI 主动推进目标，而不是被细碎规则束缚。

这个 Loop（循环）追求的是探索、迭代和自我改进。你可以阅读本文件夹内的所有材料，也可以根据任务需要主动创建、更新、整理记录文件。

## 0. 运行心法

- 主动判断下一步，不要等用户把每一步都说死。
- 优先推进目标，而不是机械执行清单。
- 可以提出新假设、新测试、新结构和新记录方式。
- 每轮都要留下可读记录，让下一轮能接着跑。
- 遇到不确定时，先做低成本验证，再决定是否继续。

## 0.1 运行模式

每次开始前，先查看 `01-state/loop-state.md` 的“运行模式”。

如果用户没有指定模式，默认使用自由模式。

- 自由模式：AI 自主判断继续、停止和下一步。适合探索、研究、整理、原型。
- 协作模式：AI 每轮给出继续/停止建议，由人确认后进入下一轮。适合重要但还不需要全自动的任务。
- 自动模式：按最大轮次、Done（完成）条件和停止条件自动推进。适合测试、巡检、周期报告等可验证任务。

运行模式是节奏选择，不是能力限制。即使在自由模式下，也要留下记录和证据。

## 1. 文件夹角色

- `00-alignment/`：任务特化初始化时的人机对齐材料
- `修改记录.md`：中文修改总账，记录改前、改后、原因、隐患和 git（版本控制）联动
- `01-state/`：当前目标、轮次、状态、阻塞点、下轮方向
- `02-test-data/`：测试数据、样例输入、参考材料
- `03-history/`：每轮实际发生了什么
- `04-optimization/`：AI 对 Loop（循环）本身的优化建议
- `05-git-backups/`：涉及 git（版本控制）时的状态记录
- `06-boundaries/`：少量硬边界和停止条件
- `07-outputs/`：本轮报告、测试日志、结论
- `08-agents/`：Maker（生产者）和 Tester（测试者）角色模板
- `09-target/`：本 Loop（循环）要优化的目标地址、文件夹或对象说明
- `00-human-docs/`：给人看的架构图和说明文档，不参与执行

## 2. 每轮启动

每轮开始时，先读：

1. `01-state/loop-state.md`
2. `06-boundaries/boundaries-and-stop.md`
3. `03-history/runs.md`
4. `04-optimization/optimization-log.md`

6. `09-target/verification.md`

如果这些文件不够用，可以主动读取其他文件，或创建新的辅助文件。

## 3. 每轮动作

每轮都完成一个闭环：

1. 理解当前目标和上下文
2. 自主选择最有价值的下一步
3. 执行这一步，或在权限不足时给出明确操作建议
4. 用测试、检查、对比、推理或人工可审查证据验证结果
5. 写入记录，并决定下一轮方向

动作可以小，也可以大。判断标准不是“是否保守”，而是“是否对目标最有推进价值”。

## 4. Maker / Tester 分离

如果任务复杂，优先采用 maker-checker（生产者-检查者）模式：

- Maker（生产者）：负责提出方案、修改内容、推进实现
- Tester（测试者）：负责独立评价、找问题、判断是否满足 Done（完成）

Tester（测试者）默认只评价，不直接修改。除非用户明确要求 tester 也参与修复。

如果当前工具支持 subagent（子代理），可以显式请求生成或调用 tester subagent（测试子代理）。如果当前工具不支持，就在同一轮里用“Tester 视角”进行独立复核。

优先复用固定 Tester（测试者），不要每轮临时重新定义 Tester。

固定 Tester 可以来自：

- `08-agents/tester-agent.md`
- Codex 的 `.codex/agents/tester.toml`
- Claude Code 的 `.claude/agents/tester.md`

## 5. 验证方式

验证不需要一开始写死。AI 应该根据 `09-target/target.md` 中的具体目标，在 `09-target/verification.md` 中维护本目标的验证方式。

验证可以分两类：

- 硬验证：测试命令、lint（代码检查）、文件是否存在、diff（差异）检查、截图对比。
- 软验证：基于证据判断是否足够好、是否覆盖关键问题、是否值得进入下一轮。

如果能用硬验证，就优先用硬验证；如果任务本身偏探索或研究，可以使用软验证，但要写清楚判断依据。

## 6. 记录要求

每轮结束时，尽量更新：

- `01-state/loop-state.md`
- `01-state/decisions.md`
- `01-state/open-questions.md`
- `03-history/runs.md`
- `04-optimization/optimization-log.md`
- `07-outputs/last-report.md`

记录不需要很长，但必须能回答：

- 本轮做了什么？
- 为什么这么做？
- 证据是什么？
- 现在离 Done（完成）还有多远？
- 下一轮最值得做什么？

## 7. 停止与升级

根据运行模式决定节奏：

- 自由模式：AI 可以自主决定继续、停止或建议下一轮。
- 协作模式：每轮结束后必须给人一个继续/停止建议。
- 自动模式：默认继续推进，直到触发停止条件。

通用停止条件：

- 达到 Done（完成）条件
- 达到最大轮次
- 连续两轮没有实质进展
- 触发 `06-boundaries/boundaries-and-stop.md` 中的硬边界
- 用户要求停止

停止时输出简短报告：当前状态、关键证据、下一步建议。
